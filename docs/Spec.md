## Архитектура навигации — ЦВСМ

## Принципы

1. Стартовая страница — реестр сценариев. Пользователь всегда начинает с выбора контекста.
2. После выбора сценария — рабочая область с четырьмя разделами в верхней навигации.
3. Активный сценарий всегда виден в шапке. Смена сценария — один клик без потери позиции.
4. Доступ к функциям определяется режимом аккаунта (Демонстрация / Рабочий).
5. Грузовой домен и финансовое моделирование — вкладки внутри карточки проекта, не отдельные разделы.

## Роли пользователей

В системе используется двухуровневая модель доступа, разделяющая официальный показ и повседневную работу.

1. **Официальная демонстрация**

- **Назначение:** Безопасный показ системы руководству и навигация по завершенным эталонным сценариям.
- **Возможности:** Просмотр реестра, открытие сценариев, просмотр всех вкладок аналитики, вызов pop-up «Информация о сценарии».
- **Ограничения:** Пользователь не может создавать, копировать, удалять или редактировать сценарии, а также изменять состав проектов или параметры расчёта. Интерфейс в этом режиме визуально очищен от лишних управляющих элементов.

2. **Рабочий аккаунт**

- **Назначение:** Аналитическая и проектная работа, моделирование альтернативных вариантов развития сети. Данный режим покрывает задачи ролей Аналитик и Эксперт.
- **Возможности:** Полный цикл управления сценариями (создание, копирование, удаление в архив, восстановление, окончательное удаление), шаринг ссылок, просмотр истории изменений (change log), редактирование пользовательских сценариев.

## Стартовый экран — Реестр сценариев

**URL:** `/`
**Назначение:** Выбор рабочего контекста перед входом в систему.

**Сущность сценария:**
Сценарий — это системный или пользовательский набор обсчитанных проектов на заданном горизонте планирования, предназначенный для анализа влияния выбранного набора проектов на сеть и экономику с учетом приоритетности проектов и этапов реализации.

**Обязательные атрибуты сценария:**

- Название
- Описание
- Горизонт планирования (начальный и конечный год)
- Состав проектов (ключевой обязательный атрибут)
- Автор
- Тип сценария: системный / пользовательский
- История изменений (change log)

_Примечание: На текущем этапе детальная расчетная логика и статусы вычислений не описываются и остаются вне рамок этой версии документа._

**Системный сценарий («Утверждённый»):**

- Является эталонным утвержденным сценарием развития сети.
- Предназначен для просмотра и официальной демонстрации.
- ✅ Нельзя редактировать напрямую.
- ✅ Нельзя архивировать.
- ✅ Нельзя удалять.
- ✅ Можно открыть.
- ✅ Можно скопировать.
- ✅ Можно использовать как основу для пользовательского сценария.
- При нажатии «Редактировать сценарий» система не редактирует его напрямую, а предлагает создать копию сценария.
- Включает 8 зафиксированных проектов:
  1. ВСМ Москва – Санкт-Петербург (Введённый)
  2. ВСМ Москва – Екатеринбург
  3. ВСМ Москва – Адлер
  4. ВСМ Москва – Минск (Международный)
  5. ВСМ Москва – Рязань
  6. СМ Москва – Белгород
  7. СМ Москва – Брянск
  8. СМ Москва – Ярославль

**Структура реестра:**

- Две вкладки: «Активные» (по умолчанию) и «Архив» (аналог корзины для рабочего аккаунта).
- Шапка страницы /scenarios: только логотип РЖД + название системы + иконка пользователя (без глобальной навигации).
- 7 демонстрационных сценариев для прототипа с разным составом проектов.

**Пользовательский сценарий:**

- ✅ Редактируется напрямую.
- ✅ Внутри него можно менять состав проектов.
- ✅ Внутри него можно сохранять ranking-конфигурации.
- ✅ Можно архивировать.
- ✅ Можно восстанавливать из архива.

---

### Логика состава проектов в пользовательском сценарии

- проекты, унаследованные из системного состава, можно включать и выключать;
- проекты, добавленные вручную через импорт или ручной ввод, также можно включать и выключать;
- включение/выключение является частью сценарного состава, а не отдельным внешним режимом редактирования;
- для пользовательского сценария не нужен промежуточный шаг через кнопку «Редактировать сценарий».

---

### Судьба вручную добавленных проектов

- если вручную добавленный проект выключен, пользователь может вернуть его только в рамках активной сессии сценария;
- после закрытия сценария выключенные вручную добавленные проекты исчезают;
- это правило явно описано в текстах уведомлений и предупреждений.

---

### Закрытие сценария

Закрытием сценария считается выход из его контекста:

- переход обратно в реестр сценариев;
- открытие другого сценария;
- завершение текущей рабочей сессии.

Закрытие сценария триггерит проверку временных объектов текущей сессии.

---

### Session-only сущности внутри сценария

- compare-набор живет только внутри активной сессии сценария;
- при закрытии сценария compare-набор очищается;
- временно выключенные вручную добавленные проекты удаляются из сценарного контекста после завершения сессии;
- ranking-конфигурации при этом не являются временными и сохраняются внутри пользовательского сценария.

---

### Создание сценария

- создание сценария открывает форму с обязательными параметрами сценария: название, описание, горизонт планирования;
- после создания сценария пользователь попадает внутрь нового сценария;
- первым экраном нового сценария всегда становится карта сети;
- сценарий создается как пользовательский.

---

### Копирование сценария

- копирование возможно из реестра сценариев и изнутри системного сценария;
- копия создается как пользовательский сценарий;
- копия наследует исходный состав проектов и базовые параметры сценария;
- после подтверждения копирования пользователь открывает уже копию, а не остается внутри системного сценария;
- новая копия имеет собственный scenarioId и собственный lifecycle.

---

### Share-сценарий

- share применяется к сценарию как к сущности;
- пользователь может отправить доступ по e-mail;
- у приглашения должен быть понятный статус;
- пользователь должен видеть, кому сценарий уже открыт;
- пользователь должен иметь возможность отозвать доступ;
- share не существует как одноразовый action без управления дальнейшим состоянием.

---

### Archive / Restore

- архивирование доступно только для пользовательского сценария;
- архивирование убирает сценарий из активного реестра, но не уничтожает его;
- у архивного сценария должен быть понятный способ восстановления;
- сценарий можно восстановить из архива обратно в активный список.

---

### Change log

- change log относится к сценарию как к корневой сущности;
- change log фиксирует как минимум:
  - создание сценария;
  - копирование сценария;
  - изменение состава проектов;
  - добавление проекта;
  - выключение / включение проекта;
  - архивирование;
  - восстановление;
  - изменение прав доступа / share;
  - сохранение ranking-конфигурации;
- change log должен быть читаемым и понятным по автору, времени и сути изменения.

---

### Действия в строке сценария:

В демонстрационном режиме интерфейс визуально очищен от управляющих действий, чтобы исключить случайные клики во время официального показа.

- **Для режима «Официальная демонстрация»:**
  - Открыть сценарий
  - Информация о сценарии (read-only сводка)

- **Для режима «Рабочий аккаунт»:**

  **Для системного сценария:**
  - Открыть
  - Скопировать
  - Поделиться
  - Просмотреть change log

  **Для пользовательского сценария:**
  - Открыть
  - Поделиться
  - Архивировать
  - Просмотреть change log

Прямое «Редактировать» в реестре как отдельный сценарный режим отсутствует. Редактирование пользовательского сценария происходит уже после входа в него через действия над составом и данными.

**Логика управления (Рабочий аккаунт):**

- **Копирование:** Копия наследует все параметры (состав проектов, горизонт, настройки ранжирования). Используется для построения альтернативных сюжетов (оптимистичный / реалистичный / пессимистичный). При изменении горизонта планирования в копии система инициирует пересчёт данных.
- **Поделиться:** Генерация прямой ссылки или отправка по e-mail. Шаринг не меняет права доступа — получатель авторизуется и видит сценарий согласно правам своего аккаунта.
- **Архив и Удаление:** Удаление перемещает сценарий во вкладку «Архив». Из архива сценарий можно восстановить или удалить окончательно (требуется отдельное подтверждение).
- **История изменений:** Change log фиксирует кто, когда и какие поля изменял.

**Pop-up «Информация о сценарии»:**
Краткая сводка для ориентации в реестре, не является формой редактирования. Отображает: Название, Описание, Тип, Автор, Горизонт, Кол-во проектов, Дата изменения.

### Диалог «Создать сценарий» (Рабочий аккаунт)

**Поля формы:**

1. Название (required)
2. Описание (optional)
3. Горизонт планирования (required)
4. Состав проектов (по умолчанию 8 системных, возможность редактирования и добавления новых через GeoJSON/координаты).

_Логика:_ После создания выполняется базовый расчет, и пользователь автоматически перенаправляется на страницу «Карта сети» нового сценария.

---

### Базовый flow

1. Пользователь открывает реестр сценариев.
2. Выбирает нужный сценарий (кликом по строке).
3. Система открывает рабочую область выбранного сценария.
4. Первая страница сценария всегда — «Карта сети».

- Детальные пользовательские сценарии по работе с реестром, переключению сценариев и экспорту вынесены в отдельный файл [TsVSM-3.5-flows.md](file:///Users/igrlebed/Desktop/%D0%94%D0%BE%D0%BA%D1%83%D0%BC%D0%B5%D0%BD%D1%82%D0%B0%D1%86%D0%B8%D1%8F/%D0%A6%D0%92%D0%A1%D0%9C/TsVSM-3.5-flows.md).

## Рабочая область

Внутри сценария зафиксирован единый постоянный каркас шапки:

[Логотип] Карта сети | Проекты | Сравнение | Ранжирование [Сценарий: Утверждённый ▼] [Аватар]

### Шапка (всегда видна)

- **Навигация** — четыре основных раздела всегда доступны: Карта сети, Проекты, Сравнение, Ранжирование.
- Экспорт не является отдельным глобальным разделом в шапке.
- Быстрый экспорт — это локальное действие внутри конкретного раздела.
- Расширенный экспорт — это отдельный модуль, который открывается только из локального экспорта и не существует как пятый постоянный раздел верхней навигации.

### Правила открытия и смены сценария

- При открытии сценария из реестра, после создания сценария и после копирования сценария пользователь всегда попадает на `/map` (Карта сети).
- При быстрой смене сценария через overlay в шапке изнутри рабочей области система пытается сохранить текущий раздел (/map, /projects, /compare, /ranking).
- Если сохранение текущего раздела невозможно или невалидно в новом контексте — происходит fallback на `/map`.
- **Активный сценарий** — кликабельный блок в правой части шапки.
  - При клике открывается компактный **overlay / popover** для быстрой смены сценария.
  - В overlay пользователь может быстро выбрать другой сценарий из списка или перейти в полный реестр сценариев.
  - Overlay является вспомогательным слоем поверх текущего экрана и не является отдельным полноэкранным модулем.
- **Сохранение контекста** — при смене сценария система сохраняет текущие пользовательские настройки отображения. Несохраненные изменения на данном этапе не рассматриваются как отдельный сценарий.

Каноническая точка входа в рабочую область — «Карта сети».

## State 1 — Default

По умолчанию на карте отображаются все проекты, включенные в текущий сценарий. Карта работает как верхнеуровневый обзор состава сценария и предназначена в том числе для демонстрации динамики развития сети на горизонте планирования.

### KPI-стрип

Агрегат по всем проектам активного сценария. 6 показателей:

- Прирост ВВП (трлн ₽)
- Население в зоне влияния (млн чел.)
- Пассажиропоток (млн чел./год)
- Протяжённость сети (км)
- Потребный парк (составов)
- Инвестиции (млрд ₽)

### Панель управления картой

- Переключатель: **Карта** | Список (переход в раздел Проекты)
- Кнопка «Показать гант» → открывает поп-ап с гантом
- Кнопка «Добавить проект» — активна в пользовательском сценарии и в режиме редактирования пользовательской копии системного сценария; при системном сценарии сначала предлагается создать копию.
- Кнопка «Экспортировать» → быстрый экспорт текущего экрана
- **Кнопка «Информация о сценарии»**: открывает правую боковую панель с read-only сводкой по активному сценарию. Редактирование сценария через эту панель не выполняется. Панель сдвигает фрейм карты.

### Слайдер годов

- Расположен под картой.
- Диапазон слайдера определяется горизонтом планирования активного сценария.
- Слайдер задаёт выбранный годовой срез, который привязан как к карте, так и к KPI-стрипу.
- На карте активными отображаются проекты, у которых год ввода меньше или равен выбранному значению слайдера.
- Проекты с более поздним годом ввода остаются на карте в полупрозрачном состоянии.
- KPI-стрип пересчитывается под выбранный годовой срез.
- Проекты, не вошедшие в выбранный годовой срез, не сохраняются в compare basket.
- Слайдер не является декоративным timeline: он управляет состоянием видимости проектов и обновляет показатели.

### Карта

- Тайлы: CartoDB Positron. Маршруты: полилинии по цветам. Контролы: зум, фит-баундс.

**Тематические слои:**
Цветовая дифференциация маршрутов на карте по выбранному параметру.
Доступные слои:

- По типу проекта (ВСМ / СМ / ВСМ Международный / ВСМ Введённый) — цвета из дизайн-системы
- По статусу (Введён / В разработке / Черновик)
- По году ввода (градиент по временной шкале)
  Переключение слоёв — кнопка «Слои» в панели управления картой, открывает дропдаун с чекбоксами.
  Для прототипа: тематические слои реализуются через цветовые бейджи маршрутов, реальная тайловая стилизация — **_backlog v2_**.

## State 2 — Проект выбран

### Боковая панель (открывается справа)

- Панель открывается при клике на маршрут и сдвигает рабочую область карты, не перекрывая её.
- Панель фиксируется справа и смещает карту на 360px.
- Показывает только базовую информацию о проекте:
  - Название проекта
  - Тип проекта
  - Статус происхождения: утверждённый / добавленный
  - Статус реализации (если доступен в данных)
  - Начальная и конечная точки маршрута
  - Протяжённость
  - Год запуска
- В панели есть кнопка «Открыть карточку проекта» для перехода в детальную карточку проекта.

## State 3 — Режим сравнения (на карте)

- Множественный выбор проектов доступен только в режиме сравнения.
- Выбранные проекты отображаются в боковой панели стеком, друг под другом.
- Режим сравнения на карте нужен для быстрого визуального фокуса и предварительного сопоставления проектов.
- Для полного сравнения пользователь переходит в отдельный модуль «Сравнение».
- Переход в модуль сравнения использует уже выбранный на карте набор проектов.
- Если проект перестаёт входить в выбранный годовой срез, он автоматически удаляется из compare basket.

## State 4 — Режим редактирования сценария (на карте)

- Кнопка «Редактировать сценарий» существует только в системном сценарии.
- При нажатии эта кнопка не запускает прямое редактирование системного сценария.
- Система показывает уведомление с предложением создать пользовательскую копию системного сценария.
- После подтверждения создаётся пользовательская копия сценария.
- Дальнейшие изменения состава проектов выполняются уже в копии, а не в исходном системном сценарии.
- Кнопка «Добавить проект» активна только в режиме редактирования пользовательской копии сценария.
- Диалог добавления поддерживает два основных способа добавления проекта: загрузка файла геометрии и ручной ввод.
- Рисование на карте остаётся в backlog и не является обязательным активным способом добавления на текущем этапе.
- После успешного добавления проекта система показывает уведомление.
- Новый проект сразу отображается на карте и становится доступен для взаимодействия наравне с остальными.
- Пользователь остаётся в контексте текущего экрана «Карта сети», без перехода в другой раздел.

## State 4a — Пользовательский сценарий

- В пользовательском сценарии состав проектов меняется напрямую, без дополнительного шага через кнопку «Редактировать сценарий».
- Унаследованные из системного состава проекты можно включать и выключать.
- Вручную добавленные через импорт или ввод данные проекты также можно включать и выключать.
- Если вручную добавленный проект выключен, он может быть возвращён только в рамках активной сессии сценария.
- После закрытия сценария выключенные вручную добавленные проекты исчезают.
- Это поведение фиксируется в уведомлениях и предупреждениях внутри сценария.

### Логика редактирования проекта

Редактирование проекта выполняется через pop-up. Pop-up открывается в контексте конкретного блока / тематической группы данных, а не как единая бесконечная форма на все вкладки сразу. Pop-up должен давать полный фокус на задаче, пользователь не должен искать редактируемые параметры по разным вкладкам.

✅ **Нельзя редактировать:**

- название проекта;
- базовый год запуска.

✅ **Можно редактировать:**

- параметры ПИР;
- параметры СМР;
- параметры ввода / этапности;
- редактируемые исходные технические вводные;
- редактируемые финансовые допущения;
- редактируемые параметры спроса;
- редактируемые риск-параметры;
- другие значения, логически связанные с пользовательским сценарием, если они не противоречат фиксированным идентификаторам проекта.

---

### Правило пересчета после редактирования

После редактирования пересчитываются все связанные показатели, которые зависят от измененных значений. Пользователь должен видеть, что данные были изменены. На текущем этапе backend-процесс пересчета детально не описывается.

---

### Индикация измененных данных

UX-правило:

- желтая точка на вкладке показывает, что внутри вкладки есть измененные данные;
- локальный маркер возле значения или строки показывает конкретно измененный элемент;
- не используется перегруженная система статусов.

---

### Правило доступа к редактированию

- в системном сценарии проекты напрямую не редактируются;
- любая попытка редактирования внутри системного сценария должна вести к копированию сценария;
- в пользовательском сценарии проекты можно редактировать свободно.

Для карточки проекта, сравнения, ранжирования и расширенного экспорта обязательны состояния:

- `loading`
- `empty`
- `error`

Для вкладок карточки проекта эти состояния реализуются на уровне каждой отдельной вкладки.

## Раздел 3 — Сравнение

**URL:** `/compare`
**Для кого:** Аналитик, Эксперт — табличное сопоставление проектов.

✅ Модуль `/compare` является отдельным независимым разделом системы.

### Общие принципы

- Количество проектов в сравнении не ограничено.
- Baseline-проект в сравнении не используется. Отсутствует эталонный или базовый проект как опорная сущность.
- Структура сравнения полностью повторяет структуру карточки проекта. Пользователь сравнивает проекты по тем же 8 вкладкам и смысловым блокам, что уже знает из карточки.
- Основной формат Compare — таблица. В дополнение к таблице на вкладке «Обзор» допускается ограниченный набор согласованных визуализаций: одна сводная диаграмма grouped bar по сопоставимым нормированным метрикам (пассажиропоток, протяжённость, инвестиции, эффект на ВВП). Никаких других chart-mode и переключателей режимов нет.
- Compare предназначен для сопоставления проектов. Не рассчитывает итоговый приоритет и не подменяет модуль ранжирования.

### Поведение таблицы сравнения

- Sticky только шапка таблицы.
- Первый столбец содержит название метрики / параметра.
- Остальные столбцы — выбранные проекты.
- Таблица поддерживает горизонтальный скролл при любом количестве проектов. Не вводится жесткий лимит на количество колонок.

### Логика дельты и подсветки

- Если выбрано ровно 2 проекта, система показывает дельту по сравниваемым значениям.
- Если выбрано больше 2 проектов, отдельная дельта не строится.
- При сравнении 3 и более проектов система подсвечивает лучший и худший показатель по строке цветом.
- Цветовая подсветка используется только для выделения экстремумов. Не используется сложная heatmap-логика.

### Логика хранения compare-набора

- Compare-набор сохраняется только внутри активной сессии сценария.
- После завершения сессии compare-набор не сохраняется как постоянная сущность.

### Логика возврата из сравнения

- Возврат обратно выполняется единообразно через breadcrumbs или верхнюю навигацию.
- Compare не является тупиковым экраном.

### Логика экспорта

- Быстрый экспорт текущего состояния сравнения: Excel / PDF / PNG.
- Отдельный переход в общий модуль экспорта для детальной настройки выгрузки.
- Compare использует общую для системы модель экспорта, не имеет уникальных отдельных правил.

## Раздел 4 — Ранжирование

**URL:** `/ranking`
**Для кого:** Аналитик, Эксперт — приоритизация проектов по согласованным критериям и весам.

✅ Ranking предназначен для приоритизации проектов. Compare предназначен для табличного сопоставления проектов. Это разные модули с разной целью, ranking не повторяет интерфейс compare.

### Методология ранжирования

- В текущей версии используется только один метод ранжирования — TOPSIS.
- Выбор между несколькими методами пользователем не предусмотрен.
- Метод AHP не является активным в текущей версии и реализуется на следующих этапах.
- Анализ чувствительности (sensitivity-analysis) не реализуется в текущей версии.

### Согласованные критерии ранжирования

1.  **Транспортные критерии:**
    - Перспективный пассажиропоток
    - Индуцированный пассажирооборот
    - Сокращение времени поездки

2.  **Дополнительная пропускная способность**

3.  **Критерии капиталоемкости:**
    - Удельные инвестиции в проект на 1 пассажира
    - Удельные инвестиции в проект на 1 км

4.  **Критерии социально-экономической эффективности:**
    - Коэффициент социально-экономической эффективности проекта
    - Коэффициент эффективности бюджетных вложений
    - Доля бюджетного финансирования проекта
    - Срок окупаемости бюджетных средств
    - Суммарный вклад в ВВП РФ от реализации проектов

### Логика весов

- В системе существует предустановленный согласованный набор весов.
- Пользователь может изменять веса вручную.
- Ручная настройка весов является вторичной функцией интерфейса, не доминирует и раскрывается по явному действию «Настроить веса».
- На первом плане пользователь видит уже заданную конфигурацию.

### Правило суммы весов

- Сумма весов должна быть строго равна 100%.
- Если пользователь меняет веса и сумма перестает быть равной 100%, система показывает явное уведомление о несоответствии.
- Система предлагает автоматически нормализовать веса до 100%.
- Расчет ранжирования не выполняется пока сумма весов не приведена к 100%.

### Результат ранжирования

Пользователь в результате видит:

- список проектов;
- итоговый балл;
- ранг;
- вклад каждого критерия в итоговый результат.

✅ Результат ранжирования является объяснимым.

### Сохранение ranking-конфигурации

- Конфигурации ранжирования сохраняются внутри текущего сценария.
- Сохраняемая конфигурация включает: набор критериев, веса, результат ранжирования.
- Ranking-конфигурация является частью сценарного контекста.

### Логика экспорта

- Быстрый экспорт текущего состояния ранжирования: Excel / PDF / PNG.
- Отдельный переход в общий модуль экспорта для детальной настройки выгрузки.
- Ranking использует общую для системы модель экспорта, не имеет уникальных отдельных правил.

## Экспорт

В системе разведены два уровня экспорта данных:

### 1. Быстрый экспорт

- **Назначение:** Локальное действие на конкретной открытой странице для создания быстрого слепка текущего экрана.
- **Доступность:** Кнопка «Экспортировать» в правом верхнем углу каждого экрана внутри сценария.
- **Форматы:** Excel / PDF / PNG.
- Быстрый экспорт не заменяет отдельный экспортный модуль.

### 2. Модуль экспорта

- **Назначение:** Отдельный расширенный контур экспорта для настройки условий выгрузки.
- **Доступность:** Открывается только из локального экспорта текущего раздела, не является постоянным разделом в шапке.
- **Возможности:** Позволяет собирать таблицы, настраивать состав выгрузки, формировать пакет PNG по нужным разделам и т.д.
- **Содержит конструктор отчёта:**
  - Выбор разделов для включения в отчёт (чекбоксы)
  - Выбор проектов
  - Настройка формата (PDF / Excel / PowerPoint)
  - Настройка шаблона (корпоративный РЖД / базовый)
- **Важно:** Формат PowerPoint не реализован в текущей версии
  - Область предпросмотра
- Для прототипа: весь модуль — заглушка с макетом интерфейса и текстом «[Конструктор отчётов — в разработке]»
- Кнопка «Сформировать отчёт» активна, но показывает toast «Функция будет доступна в следующей версии»

## Архитектура и маршрутизация

### 2.1. Уровень экрана
Сопоставление сценариев — **экран реестрового уровня**, а не подрежим рабочей области сценария. Он не имеет верхней навигации из 4 разделов (`Карта / Проекты / Сравнение / Ранжирование`).

### 2.2. Маршрут
```
/registry/compare?scenarios=id1,id2,id3&baseline=id1
```
- URL-driven: набор сценариев и baseline хранятся в query-параметрах.
- Поддерживает прямую ссылку (shareability) без сохранения состояния в БД.
- При открытии по прямой ссылке система валидирует доступ к каждому `scenarioId`; при отсутствии доступа — `denied`-state.

### 2.3. Шапка экрана
Упрощённый каркас (не рабочая область сценария):
```
[Логотип]  Реестр сценариев / Сопоставление сценариев  [Аватар]
```
- Нет переключателя активного сценария.
- Нет глобальных 4 разделов.
- Breadcrumb ведёт в реестр (`/`).

## Точки входа (Entry Points)

### 3.1. Основной — из реестра сценариев (`/`)
- В таблице реестра появляется режим множественного выбора: чекбокс в строке (виден по ховеру или при активации режима `Выбрать`).
- При выборе 2+ сценариев появляется **sticky bottom bar**: `Сопоставить (N)`.
- Клик по кнопке открывает шаг выбора baseline, затем переход на `/registry/compare`.

### 3.2. Вторичный — из dropdown активного сценария в шапке
- В overlay выбора сценария (`Conventions.md §1`, `Flow.md §5`) добавить действие: `Сопоставить с другими`.
- Открывает overlay выбора дополнительных сценариев поверх текущего контекста.
- Текущий сценарий предвыбран и заблокирован для удаления из набора.
- Пользователь добавляет 1–3 сценария → кнопка `Открыть сопоставление` → переход на `/registry/compare` с предзаполненным набором.

### 3.3. Локальный — внутри рабочей области сценария
- В шапке рабочей области, рядом с блоком активного сценария (или в kebab-меню действий сценария), добавить CTA: `Сопоставить с другими`.
- **Критично:** действие **не открывает** сопоставление inline или в модалке. Система уводит пользователя на отдельный маршрут `/registry/compare` с предзаполненным текущим сценарием.

## Пользовательские потоки

### 4.1. Флоу из реестра
1. Пользователь открывает реестр (`/`).
2. Активирует режим выбора (чекбоксы).
3. Выбирает 2–4 сценария.
4. Нажимает `Сопоставить (N)` на bottom bar.
5. Система открывает диалог выбора baseline из числа выбранных.
6. Пользователь подтверждает baseline.
7. Система валидирует статусы сценариев (см. §6.3).
8. Переход на `/registry/compare?scenarios=...&baseline=...`.
9. Пользователь анализирует KPI, состав и временно́й профиль.
10. Доступные действия из результата:
    - `Открыть сценарий` — переход в `/map` выбранного сценария.
    - `Сделать активным` — смена текущего сценария с fallback на `/map`.
    - `Экспорт` — быстрый экспорт текущего сопоставления (Excel/PDF/PNG).
    - `Поделиться` — копирование URL с набором.
    - `Вернуться в реестр` — переход на `/`.

### 4.2. Флоу изнутри сценария
1. Пользователь находится в рабочей области сценария (например, `/map` или `/projects/:id`).
2. Нажимает `Сопоставить с другими` в шапке / kebab-меню.
3. Система открывает overlay выбора сценариев:
    - текущий сценарий предвыбран (chip заблокирован);
    - доступны только сценарии, к которым у пользователя есть доступ.
4. Пользователь добавляет 1–3 сценария.
5. Нажимает `Открыть сопоставление`.
6. Система уводит на `/registry/compare?scenarios=...&baseline=текущий`.
7. **Критично:** активный сценарий пользователя **не меняется**. Сопоставление — аналитический режим, а не переключение контекста.
8. Пользователь завершает анализ и нажимает `Вернуться в сценарий` — возврат на `/map` исходного сценария.
9. Альтернатива: нажимает `Открыть сценарий` на одном из сопоставленных — переход в рабочую область этого сценария.

## Атрибутный состав экрана

### 5.1. Верхняя панель (header area)

| Элемент | Тип | Описание |
| :--- | :--- | :--- |
| Breadcrumb | Навигация | `Реестр сценариев / Сопоставление сценариев` |
| Chips сценариев | Интерактивные метки | Цветные chips с названием и типом (системный / пользовательский). Baseline помечен бейджем «Базовый». Клик по крестику удаляет сценарий из набора (если >2 остаётся). |

| `Экспорт` | Кнопка | Быстрый экспорт текущего вида. |

### 5.2. KPI-слой (summary cards)

Агрегированные показатели по каждому сценарию в наборе. Для MVP — 8 карточек:

| KPI | Единица | Дельта от baseline |
| :--- | :--- | :--- |
| NPV (ЧДД) | млрд ₽ | ± млрд ₽, % |
| IRR (ВНД) | % | ± п.п. |
| Инвестиции (CAPEX) | млрд ₽ | ± млрд ₽, % |
| Пассажиропоток | млн чел./год | ± млн, % |
| Прирост ВВП | трлн ₽ | ± трлн, % |
| Население в зоне влияния | млн чел. | ± млн, % |
| Срок окупаемости | лет | ± лет |
| Рейтинговый балл TOPSIS | 0–1 | ± |

**Визуализация:** grouped bar chart (нормированный, как в `backlogsfunction.md` для Compare), где каждый столбец — сценарий, цветом кодирован baseline.

### 5.3. Вкладки детального анализа (MVP)

| Вкладка | Содержание | Статус |
| :--- | :--- | :--- |
| **KPI** | Таблица 8 показателей × N сценариев + дельты + grouped bar chart | MVP |
| **Состав и отличия** | Diff-представление проектов: включён/выключен, добавлен вручную, удалён, изменены параметры. Группировка по типу проекта (ВСМ/СМ). | MVP |
| **Время́нный профиль** | Гант-сводка по сценариям: когда вводится каждый проект в каждом сценарии. Годовой срез 2030–2050. | MVP |

**Backlog v3.6+:**
- `Эффективность` — декомпозиция NPV/IRR по источникам эффекта.
- `Риски и ограничения` — агрегированная матрица рисков по сценарию (требует агрегации рисков на уровень сценария).

### 5.4. Вкладка «Состав и отличия» (детально)

Центральная вкладка. Структура:

| Колонка | Описание |
| :--- | :--- |
| Проект | Название, тип, маршрут |
| Статус в сценарии 1 | Включён / Выключен / Отсутствует |
| Статус в сценарии 2 | ... |
| ... | ... |
| Отличие | Автоматический тег: `Добавлен`, `Удалён`, `Выключен`, `Изменены параметры`, `Без изменений` |

**Логика отличий:**
- Если проект есть в baseline, но отсутствует в другом → `Удалён`.
- Если проекта нет в baseline, но есть в другом → `Добавлен`.
- Если проект есть в обоих, но статус «Выключен» в одном → `Выключен`.
- Если проект есть в обоих, но изменены ключевые параметры (год запуска, длина, CAPEX) → `Изменены параметры` (hover показывает diff).

## Логика выбора сценариев и ограничения

### 6.1. Жёсткие ограничения
- **Минимум:** 2 сценария.
- **Максимум:** 4 сценария. При попытке выбрать 5-й — тултип «Можно сопоставить не более 4 сценариев» и блокировка чекбокса.
- **Baseline:** обязателен. Без выбора baseline кнопка `Открыть сопоставление` заблокирована.

### 6.2. Матчинг проектов между сценариями
Для построения diff проектов используется **стабильный `projectId`**:
- **Системные проекты** (8 утверждённых) имеют одинаковый `projectId` во всех сценариях (`approved`, `base`, пользовательских копиях).
- **Пользовательские проекты**, добавленные вручную, наследуют `parentScenarioId` в поле `originScenarioId`. При копировании сценария `projectId` пользовательских проектов меняется, но `originScenarioId` сохраняется для трассировки.

**Diff-алгоритм:**
1. Union всех `projectId` из всех сценариев набора.
2. Для системных проектов — прямое сравнение по `projectId`.
3. Для пользовательских — fuzzy-match по `originScenarioId` + названию + маршруту. Если совпадение >80% — считаем тем же проектом, иначе — новая сущность.

### 6.3. Гейты по статусам сценария
При выборе сценария в набор система проверяет:

| Статус сценария | Поведение |
| :--- | :--- |
| `actual` | ✓ Разрешён |
| `in_calculation` | ⚠️ Предупреждение: «Сценарий в расчёте, данные могут измениться». Разрешён, но baseline не рекомендуется. |
| `stale_after_changes` | 🚫 Блокировка с CTA: «Пересчитать сценарий перед сопоставлением» или «Всё равно сопоставить» (явное подтверждение). |
| `calculation_error` | 🚫 Блокировка: «Сценарий содержит ошибку расчёта, сопоставление невозможно». |
| `not_calculated` | 🚫 Блокировка: «Сценарий не рассчитан». |
| `empty` (0 проектов) | 🚫 Блокировка: «Сценарий не содержит проектов». |

## Состояния экрана (StatesMatrix integration)

| Состояние | Когда | UI-проявление |
| :--- | :--- | :--- |
| `loading` | Загрузка 2–4 сценариев параллельно | Skeleton карточек KPI + skeleton таблиц. Прогрессивная загрузка: по мере готовности каждого сценария заполняется его колонка. |
| `empty` | Набор содержит <2 сценариев после удаления из chips | Empty-state: «Добавьте минимум 2 сценария для сопоставления», CTA «Вернуться в реестр». |
| `error` | Ошибка загрузки одного из сценариев | Error-state с указанием, какой сценарий недоступен. CTA «Удалить из набора и продолжить» или «Повторить». |
| `partial` | Один из сценариев загружен с ошибкой части данных | Предупреждение в шапке: «Данные по [Название] загружены частично». Таблица показывает `—` в ячейках. |
| `ready` | Все сценарии загружены | Основной интерфейс. |
| `denied` | Нет доступа к одному из сценариев по прямой ссылке | 403-state со ссылкой на реестр. |

## Экспорт

**Быстрый экспорт** (локальное действие экрана):
- Форматы: Excel, PDF, PNG.
- Содержимое: текущая вкладка (KPI / Состав / Время́нный профиль) + шапка с перечнем сценариев.
- Не открывает общий модуль экспорта с конструктором.

**Расширенный экспорт** (backlog v3.6+):
- Возможность собрать пакетный отчёт по сопоставлению: KPI + diff + графики.

## Backlog и риски

### 10.1. Backlog v3.6+
- Вкладка `Эффективность` — декомпозиция NPV по источникам.
- Вкладка `Риски и ограничения` — агрегация рисков на уровень сценария.
- Сопоставление по годовым срезам (не только итоговые KPI, но и динамика 2030→2050).
- Сохранение «избранных наборов сопоставления» в профиле пользователя.
- Сопоставление не по сценариям целиком, а по «что если» (what-if) с подменой одного проекта.

### 10.2. Риски реализации
| Риск | Митигация |
| :--- | :--- |
| Производительность: параллельная загрузка 4 сценариев с полными данными | Прогрессивная загрузка колонок; кэширование сценариев в `ProjectsContext` |
| Матчинг пользовательских проектов ломается при переименовании | Использовать `originScenarioId` + геометрия маршрута как fallback |
| Пользователь путает `/compare` (проекты) и `/registry/compare` (сценарии) | Жёсткая терминология, разные URL, разные шапки, onboarding-тур |
| Baseline с `stale_after_changes` даёт ложные дельты | Гейт + явное предупреждение перед открытием |

*Дата: 05.05.2026*
Назначение файла: детально зафиксировать пользовательские сценарии системы ЦВСМ и исключить неоднозначность между архитектурой, навигацией и поведением интерфейса.
Глобальные правила для всех flow:
Стартовая точка системы — реестр сценариев /.
После открытия сценария из реестра, после создания сценария и после копирования сценария пользователь всегда попадает на /map.
Внутри рабочей области есть только 4 постоянных раздела: Карта сети, Проекты, Сравнение, Ранжирование.
Экспорт не является отдельным постоянным разделом верхней навигации.
Быстрый экспорт — локальное действие текущего раздела.
Расширенный экспорт — отдельный модуль, который открывается только из локального экспорта текущего раздела.
Информация о сценарии — read-only сводка и не используется как режим редактирования.
При быстрой смене сценария через overlay система пытается сохранить текущий раздел; если это невозможно — выполняется fallback на /map.
Системный сценарий нельзя редактировать напрямую; редактирование возможно только через создание пользовательской копии.
Для сценария, проекта и аналитических блоков используются статусы расчёта. Канонический набор и UI-лейблы — `docs/Glossary.md §1` (`not_calculated`, `in_calculation`, `actual`, `stale_after_changes`, `calculation_error`). Любые отклонения от формулировок Glossary запрещены.
Для тяжелых экранов и блоков обязательны состояния loading, empty, error.
В ранжировании используется только TOPSIS; пользователь не выбирает метод; анализ чувствительности не реализуется.

### 1. Открытие реестра сценариев

**Цель пользователя:** выбрать рабочий контекст перед входом в систему.

**Предусловия:** пользователь авторизован и находится на стартовом экране системы.

**Шаги:**

Пользователь открывает систему. Система открывает экран реестра сценариев /. Данные: список активных сценариев, режим аккаунта пользователя.

Пользователь просматривает вкладку Активные. Система показывает таблицу сценариев с названием, типом, количеством проектов, датами, автором и доступными действиями по строке. Данные: scenarioId, type, projectCount, createdAt, updatedAt, author.

Пользователь при необходимости переключается на вкладку Архив. Система показывает архивные пользовательские сценарии с действиями восстановления и окончательного удаления, если это допускает режим аккаунта. Данные: archived scenarios, restore permissions.

Пользователь использует поиск или фильтрацию. Система обновляет таблицу без выхода со страницы. Данные: search query, active/archive tab, filtered result set.

**Альтернативные пути:**

В режиме Официальная демонстрация пользователь может только открыть сценарий и посмотреть Информацию о сценарии.

В режиме Рабочий аккаунт пользователь получает действия создания, копирования, share, архива и change log согласно типу сценария.

**Где пользователь может сломаться:**

Не понимает, почему для системного сценария нет архивации и удаления.

Не видит разницу между системным и пользовательским сценарием.

**Системные состояния / ответы:**

loading — skeleton строк таблицы.

empty — нет сценариев во вкладке.

error — ошибка загрузки списка сценариев.

### 2. Открытие сценария из реестра

**Цель пользователя:** войти в нужный сценарий и начать анализ.

**Предусловия:** пользователь находится в реестре сценариев, хотя бы один сценарий доступен.

**Шаги:**

Пользователь кликает по строке сценария или по действию Открыть. Система принимает выбранный scenarioId. Данные: scenarioId, scenario type, user permissions.

Система открывает рабочую область выбранного сценария. Первым экраном всегда становится /map. Данные: scenario context, default route=/map.

Система загружает карту, KPI-стрип и состав проектов текущего сценария. Данные: project set, horizon, KPI aggregate, layer defaults.

Пользователь начинает работу внутри сценария уже из контекста карты сети.

**Альтернативные пути:**

Если сценарий был открыт из режима демонстрации, пользователь попадает в тот же /map, но без управляющих действий редактирования.

**Где пользователь может сломаться:**

Ожидает возврат в “последний раздел”, но система всегда приводит его на карту при первичном входе в сценарий.

**Системные состояния / ответы:**

loading — скелетон карты и KPI.

error — сценарий не загрузился или недоступен.

### 3. Создание сценария

**Цель пользователя:** создать пользовательский сценарий для моделирования альтернативного набора проектов.

**Предусловия:** пользователь находится в режиме Рабочий аккаунт.

**Шаги:**

Пользователь нажимает + Создать сценарий в реестре. Система открывает диалог создания сценария. Данные: mode=working, create permission.

Пользователь заполняет Название, при необходимости Описание, задает горизонт планирования и состав проектов. Система валидирует обязательные поля. Данные: name, description, startYear, endYear, project set.

Пользователь подтверждает создание. Система создает новый пользовательский сценарий, сохраняет scenarioId и запускает базовый расчет. Данные: new scenarioId, lifecycle=user, calcStatus=в расчете.

Система переводит пользователя внутрь нового сценария на /map. Данные: route=/map, active scenario context.

Пока расчет не завершен, система показывает состояние в расчете на уровне сценария и модулей, а не подставляет нули.

**Альтернативные пути:**

Пользователь отменяет диалог — сценарий не создается.

Если обязательные поля не заполнены, кнопка подтверждения заблокирована.

**Где пользователь может сломаться:**

Не понимает, почему после создания данные не готовы мгновенно.

Указывает некорректный горизонт планирования.

**Системные состояния / ответы:**

validation error — пустое название, некорректный горизонт.

in calculation — расчет после создания.

error — сценарий не удалось создать.

### 4. Копирование сценария

**Цель пользователя:** быстро создать пользовательскую копию системного или пользовательского сценария для альтернативного анализа.

**Предусловия:** сценарий доступен в реестре или открыт внутри рабочей области.

**Шаги:**

Пользователь выбирает действие Скопировать. Система открывает подтверждение копирования. Данные: source scenarioId, source type, user permissions.

Система сообщает, что копия наследует горизонт, состав проектов и сценарный контекст исходного сценария. Данные: inherited horizon, inherited project set.

Пользователь подтверждает действие. Система создает новый пользовательский сценарий с новым scenarioId и собственным lifecycle. Данные: new scenarioId, source scenarioId.

Система открывает уже копию, а не оставляет пользователя в исходном сценарии. Первый экран копии — /map.

Если копия требует пересчёта после изменения параметров, система помечает статус данных как `in_calculation` или `stale_after_changes` (по Glossary). Перечень полей-триггеров перевода в `stale_after_changes` — `docs/Conventions.md §4`.

**Альтернативные пути:**

Пользователь отменяет копирование — остается в текущем контексте.

**Где пользователь может сломаться:**

Не понимает, находится ли он уже в копии или все еще в системном сценарии.

**Системные состояния / ответы:**

Toast об успешном создании копии.

Ошибка копирования, если операция не завершилась.

### 5. Быстрая смена сценария через overlay

**Цель пользователя:** переключиться на другой сценарий без возврата в реестр и без потери текущего контекста.

**Предусловия:** пользователь уже находится внутри рабочей области сценария.

**Шаги:**

Пользователь нажимает на блок активного сценария в шапке. Система открывает compact overlay/popover со списком доступных сценариев. Данные: current route, current scenarioId, available scenarios.

Пользователь выбирает другой сценарий. Система пытается сохранить текущий раздел, например /projects или /ranking. Данные: current route, target scenarioId.

Если текущий раздел валиден для нового сценария, система открывает тот же раздел в контексте нового сценария. Данные: target scenario context, preserved route.

Если текущий раздел восстановить нельзя, система делает fallback на /map. Данные: fallback route=/map.

Система сохраняет пользовательские настройки отображения, которые допустимо переносить между сценариями.

**Альтернативные пути:**

Пользователь выбирает Перейти в реестр сценариев вместо прямого переключения.

**Где пользователь может сломаться:**

Не понимает, почему после переключения попал на карту, а не остался в том же разделе.

**Системные состояния / ответы:**

Overlay остается локальным слоем, а не отдельным экраном.

При недоступности сценария система показывает ошибку и не рвет текущий контекст.

### 6. Просмотр информации о сценарии

**Цель пользователя:** быстро получить read-only сводку по сценарию.

**Предусловия:** сценарий открыт или доступен в реестре.

**Шаги:**

Пользователь нажимает Информация о сценарии. Система открывает read-only pop-up или боковую панель. Данные: scenario name, type, author, horizon, projectCount, updatedAt.

Пользователь просматривает сводку. Система не показывает editable-поля и не запускает режим редактирования.

Пользователь закрывает панель и возвращается в предыдущий контекст без изменений.

**Альтернативные пути:**

В реестре это действие открывает ту же read-only сводку.

В рабочей области поведение остается тем же — просмотр, а не редактирование.

**Где пользователь может сломаться:**

Ожидает, что через эту кнопку можно менять параметры сценария.

**Системные состояния / ответы:**

Только read-only информация.

### 7. Архивирование сценария

**Цель пользователя:** убрать пользовательский сценарий из активной работы без окончательного удаления.

**Предусловия:** сценарий пользовательский, пользователь в режиме Рабочий аккаунт.

**Шаги:**

Пользователь выбирает действие Архивировать в реестре или из сценарного контекста, если оно доступно. Система открывает подтверждение. Данные: scenarioId, type=user.

Пользователь подтверждает архивирование. Система переносит сценарий из вкладки Активные во вкладку Архив. Данные: archive status, archivedAt.

Система создает запись в change log. Данные: action=archive, author, timestamp.

Пользователь больше не видит сценарий в активном списке.

**Альтернативные пути:**

Пользователь отменяет действие — сценарий остается активным.

**Где пользователь может сломаться:**

Путает архивирование с окончательным удалением.

**Системные состояния / ответы:**

Toast об успешном переносе в архив.

### 8. Восстановление сценария из архива

**Цель пользователя:** вернуть архивный пользовательский сценарий в активную работу.

**Предусловия:** пользователь находится на вкладке Архив.

**Шаги:**

Пользователь открывает Архив и выбирает действие Восстановить. Система выполняет restore. Данные: scenarioId, archive state.

Система возвращает сценарий в активный список. Данные: active state restored.

Система пишет событие в change log. Данные: action=restore, author, timestamp.

Пользователь снова видит сценарий во вкладке Активные.

**Альтернативные пути:**

Пользователь окончательно удаляет сценарий из архива, если такая операция предусмотрена.

**Где пользователь может сломаться:**

Не понимает, вернулся ли сценарий в активный список или только “разархивировался” технически.

**Системные состояния / ответы:**

Подтверждение успешного восстановления.

### 9. Share сценария

**Цель пользователя:** открыть доступ к сценарию другому пользователю без потери контроля над доступом.

**Предусловия:** пользователь находится в Рабочем аккаунте, сценарий допускает share.

**Шаги:**

Пользователь нажимает Поделиться. Система открывает окно управления доступом. Данные: scenarioId, current access list.

Пользователь вводит e-mail и подтверждает отправку доступа. Система создает приглашение и назначает ему статус. Данные: email, invite status, scenario link.

Пользователь видит список уже приглашенных пользователей и статусы их доступа. Данные: invited users, access status.

При необходимости пользователь отзывает доступ. Система обновляет права и фиксирует это в change log. Данные: revoke action, access update.

**Альтернативные пути:**

Пользователь закрывает окно без отправки приглашения.

**Где пользователь может сломаться:**

Не понимает, отправлено приглашение или нет.

Не видит, кому доступ уже открыт.

**Системные состояния / ответы:**

Статусы приглашения должны быть явными.

Ошибка доставки/создания приглашения должна быть показана явно.

### 10. Просмотр change log

**Цель пользователя:** понять, кто и когда менял сценарий.

**Предусловия:** сценарий доступен пользователю.

**Шаги:**

Пользователь выбирает Change log. Система открывает журнал изменений сценария. Данные: scenarioId, history entries.

Система показывает список событий в читаемом формате: автор, время, тип изменения, краткое описание. Данные: author, timestamp, action type, action description.

Пользователь просматривает журнал и закрывает его без изменения сценария.

**Альтернативные пути:**

Нет.

**Где пользователь может сломаться:**

Записи слишком технические и не объясняют, что реально изменилось.

**Системные состояния / ответы:**

empty — история еще пуста.

error — журнал не загрузился.

### 11. Открытие карты сети

**Цель пользователя:** получить верхнеуровневый обзор состава сценария и состояния сети.

**Предусловия:** сценарий открыт.

**Шаги:**

Пользователь открывает раздел Карта сети или попадает на него при первичном входе. Система открывает /map. Данные: scenarioId, horizon, project set.

Система показывает KPI-стрип, карту и слайдер годов. Данные: aggregate KPI, routes, selected year.

Пользователь фильтрует проекты и двигает годовой слайдер. Система обновляет видимость маршрутов и агрегированные KPI по выбранному срезу. Данные: filters, selectedYear, active projects, future projects.

Проекты с годом ввода больше выбранного года остаются на карте полупрозрачными.

Пользователь может открыть поп-ап ганта, переключить слои и перейти к выбору проекта.

**Альтернативные пути:**

В демонстрационном режиме карта доступна как первый экран без действий редактирования.

В рабочем аккаунте доступны compare-mode, добавление проекта и локальный экспорт.

**Где пользователь может сломаться:**

Не понимает, что слайдер не просто декоративный timeline, а влияет на KPI и compare-набор.

**Системные состояния / ответы:**

loading — карта и KPI грузятся.

empty — в сценарии нет активных проектов.

error — карта или проектные данные не загрузились.

### 12. Выбор проекта на карте

**Цель пользователя:** быстро посмотреть базовую информацию по проекту и перейти в карточку.

**Предусловия:** пользователь находится на /map, хотя бы один проект отображается на карте.

**Шаги:**

Пользователь кликает по маршруту проекта. Система открывает правую боковую панель и смещает карту. Данные: projectId, geometry, panel state.

Система показывает базовые атрибуты проекта: название, тип, происхождение, статус реализации, точки маршрута, протяженность, год запуска. Данные: project summary.

Пользователь нажимает Открыть карточку проекта. Система переводит его в /projects/:id. Данные: projectId, return context=/map.

**Альтернативные пути:**

Пользователь закрывает панель и остается на карте.

**Где пользователь может сломаться:**

Ожидает увидеть в панели полную аналитику карточки.

**Системные состояния / ответы:**

Панель показывает только краткие данные, а не весь аналитический контент.

### 13. Режим сравнения на карте

**Цель пользователя:** быстро собрать набор проектов для последующего полного сравнения.

**Предусловия:** пользователь находится на карте и включил compare-mode.

**Шаги:**

Пользователь включает режим сравнения. Система переводит карту в режим множественного выбора. Данные: compare mode state.

Пользователь кликает по нескольким проектам. Система добавляет их в compare-набор и показывает выбранные проекты стеком в боковой панели. Данные: compare basket, selected projectIds.

Пользователь нажимает Перейти к сравнению. Система открывает /compare с уже собранным набором проектов. Данные: compare basket context.

Если проект выходит из активного годового среза, система удаляет его из compare-набора и явно уведомляет пользователя об этом. Данные: removed projectId, reason=year slice changed.

**Альтернативные пути:**

Пользователь снимает выбор с проекта и остается на карте.

**Где пользователь может сломаться:**

Не замечает, что часть compare-набора исчезла из-за смены года или сценария.

**Системные состояния / ответы:**

Явное уведомление при частичной очистке compare-набора.

### 14. Добавление проекта в пользовательский сценарий

**Цель пользователя:** добавить новый проект в состав пользовательского сценария.

**Предусловия:** открыт пользовательский сценарий или пользователь создал копию системного сценария.

**Шаги:**

Пользователь нажимает Добавить проект на карте. Система открывает диалог добавления проекта. Данные: scenarioId, edit permission.

Пользователь выбирает способ добавления: загрузка файла геометрии или ручной ввод. Рисование на карте не используется как обязательный способ текущей версии. Данные: import mode, geometry input.

Пользователь вводит базовые параметры проекта и подтверждает создание. Система валидирует обязательные поля и геометрию. Данные: project meta, geometry, validation result.

После успешного добавления система создает проект внутри пользовательского сценария, показывает уведомление и сразу отображает проект на карте. Данные: new projectId, scenario binding.

Новый проект становится доступен для взаимодействия, перехода в карточку, compare и дальнейшего редактирования.

**Альтернативные пути:**

Если пользователь находится в системном сценарии, система сначала предлагает создать копию сценария.

Пользователь отменяет диалог — проект не добавляется.

**Где пользователь может сломаться:**

Не понимает, добавился ли проект как полноценная сущность сценария или только как временный слой карты.

**Системные состояния / ответы:**

validation error — некорректная геометрия или пустые обязательные поля.

success — проект добавлен и отображен на карте.

### 15. Открытие списка проектов

**Цель пользователя:** увидеть табличный список проектов сценария и работать с поиском, фильтрами и сортировками.

**Предусловия:** сценарий открыт.

**Шаги:**

Пользователь открывает раздел Проекты. Система открывает /projects. Данные: scenario project set.

Система показывает таблицу со столбцами: название, тип, происхождение, протяженность, годы запуска, базовые финансовые KPI. Данные: project list, table columns.

Пользователь ищет проект по названию, применяет фильтры и сортировки. Система обновляет таблицу без выхода со страницы. Данные: search query, filters, sort state.

Клик по строке открывает карточку проекта. Множественный выбор переводит пользователя в режим подготовки compare-набора. Данные: projectId or selected projectIds.

**Альтернативные пути:**

Пользователь открывает список через переключение из карты. Маршрут должен становиться /projects, а не оставаться /map.

**Где пользователь может сломаться:**

Не понимает, почему множественный выбор приводит к сравнению, а не к массовому редактированию.

**Системные состояния / ответы:**

loading — skeleton таблицы.

empty — по фильтрам ничего не найдено.

error — список не загрузился.

### 16. Открытие карточки проекта

**Цель пользователя:** перейти к детальной аналитике выбранного проекта.

**Предусловия:** проект выбран на карте или в списке.

**Шаги:**

Пользователь открывает проект из карты или списка. Система открывает /projects/:id. Данные: projectId, source context.

Система показывает шапку карточки: название, тип, происхождение, горизонт. Данные: project header data.

Система предоставляет 8 обязательных вкладок (канонические имена и порядок — `docs/Glossary.md §5`): Обзор, Исходные данные, Пассажирские перевозки, Грузовые перевозки, Инфраструктура, Финансовая модель, Социально-экономические эффекты, Участники и риски. Данные: tab set.

Пользователь переключается между вкладками и анализирует содержимое проекта.

**Альтернативные пути:**

Пользователь добавляет проект к сравнению или открывает локальный экспорт прямо из шапки карточки.

**Где пользователь может сломаться:**

Не понимает, какая вкладка уже пересчитана, а какая в `stale_after_changes`.

**Системные состояния / ответы:**

Полный перечень и поведение состояний по 8 вкладкам — `docs/StatesMatrix.md §4.5`. Минимально: `loading` (skeleton), `empty` (нет данных), `not_calculated` (расчёт не запускался), `error` (ошибка загрузки блока). Канонические лейблы — `docs/Glossary.md §1`.

### 17. Редактирование проекта

**Цель пользователя:** изменить редактируемые параметры проекта внутри пользовательского сценария.

**Предусловия:** пользователь находится в пользовательском сценарии или его копии; проект допускает редактирование.

**Шаги:**

Пользователь нажимает Редактирование в карточке проекта. Система открывает pop-up в контексте конкретного блока данных, а не общую бесконечную форму. Данные: active tab, editable group.

Пользователь меняет допустимые параметры: ПИР, СМР, этапность, технические вводные, финансовые допущения, параметры спроса, риск-параметры. Данные: editable fields.

Система валидирует значения и не дает сохранить некорректные поля. Данные: validation rules, inline errors.

Пользователь сохраняет изменения. Система запускает пересчет зависимых показателей и меняет статус данных на в расчете, а затем на актуально или ошибка расчета. Данные: changed fields, calcStatus.

Система ставит желтую точку на вкладке и локальные маркеры на измененных значениях. Данные: change markers, stale markers.

**Альтернативные пути:**

Если пользователь пытается редактировать проект в системном сценарии, система не открывает редактирование напрямую и предлагает создать копию сценария.

**Где пользователь может сломаться:**

Не понимает, почему часть значений изменилась, а часть еще пересчитывается.

**Системные состояния / ответы:**

validation error — нельзя сохранить форму.

in calculation — пересчет после сохранения.

stale after changes — старые данные больше не актуальны.

error — пересчет не завершился.

### 18. Сравнение проектов

**Цель пользователя:** таблично сопоставить проекты по тем же смысловым блокам, что и в карточке проекта.

**Предусловия:** compare-набор уже сформирован из карты или списка.

**Шаги:**

Пользователь открывает /compare. Система поднимает compare-набор активной сессии. Данные: compare basket, scenarioId.

Система показывает вкладки, идентичные карточке проекта, и строит таблицу сравнения по выбранной вкладке. Данные: project comparison data, tab structure.

Если выбрано ровно 2 проекта, система показывает дельту по сравниваемым значениям. Если проектов больше 2, дельта не строится, а подсвечиваются лучший и худший показатели по строкам. Данные: metric values, comparison mode.

Пользователь скроллит таблицу, переключает вкладки и анализирует различия.

Пользователь может выполнить локальный экспорт текущего состояния сравнения или вернуться в другой раздел по навигации/хлебным крошкам.

**Альтернативные пути:**

Если compare-набор пуст, система не показывает пустую таблицу как будто все нормально, а выводит empty-state с CTA вернуться к выбору проектов.

**Где пользователь может сломаться:**

Не понимает, почему compare-набор опустел после смены сценария или завершения сессии.

**Системные состояния / ответы:**

loading — таблица сравнения загружается.

empty — compare-набор пуст.

error — данные сравнения недоступны.

### 19. Открытие модуля ранжирования

**Цель пользователя:** приоритизировать проекты сценария по согласованным критериям.

**Предусловия:** пользователь находится внутри активного сценария.

**Шаги:**

Пользователь открывает раздел Ранжирование. Система открывает /ranking. Данные: current scenario context.

Система загружает согласованный список критериев и предустановленный набор весов. Данные: ranking criteria, default weights.

Пользователь видит панель конфигурации и область результата. Ручная настройка весов скрыта за отдельным действием Настроить веса.

**Альтернативные пути:**

Нет.

**Где пользователь может сломаться:**

Ожидает пустой экран или выбор метода ранжирования.

**Системные состояния / ответы:**

Ранжирование открывается сразу с предустановленной конфигурацией.

### 20. Изменение весов и запуск ранжирования

**Цель пользователя:** при необходимости изменить веса критериев и получить новый результат ранжирования.

**Предусловия:** открыт /ranking.

**Шаги:**

Пользователь нажимает Настроить веса. Система раскрывает редактируемый режим весов. Данные: edit mode, current weights.

Пользователь меняет веса. Система пересчитывает текущую сумму в реальном времени. Данные: changed weights, weight sum.

Если сумма весов не равна 100%, система показывает уведомление и CTA Нормализовать веса. Расчет заблокирован.

Пользователь либо исправляет веса вручную, либо использует автонормализацию. Система приводит сумму к 100%. Данные: normalized weights.

Пользователь нажимает Рассчитать ранжирование. Система запускает TOPSIS и строит результат: список проектов, балл, ранг, вклад критериев. Данные: project metrics, TOPSIS output.

Пользователь сохраняет ranking-конфигурацию внутри текущего сценария. Данные: criteria set, weights, ranking result, scenarioId.

**Альтернативные пути:**

Пользователь отменяет изменение весов — система возвращает предыдущую конфигурацию.

**Где пользователь может сломаться:**

Не понимает, почему расчет заблокирован.

Ожидает выбор метода между TOPSIS и AHP, которого в текущей версии нет.

**Системные состояния / ответы:**

`in_calculation` — расчёт выполняется.

`calculation_error` — ранжирование не рассчиталось; единственный путь — «Повторить расчёт» (см. `Conventions.md §6`).

`actual` — результат получен и закреплён в конфигурации. Канон лейблов — `Glossary.md §1`.

### 21. Быстрый экспорт

**Цель пользователя:** выгрузить текущее состояние конкретного экрана без перехода в отдельный экспортный конструктор.

**Предусловия:** пользователь находится внутри одного из рабочих разделов сценария.

**Шаги:**

Пользователь нажимает Экспортировать на текущем экране. Система открывает локальный список форматов. Данные: current module, export context.

Пользователь выбирает формат Excel, PDF или PNG. Система формирует быстрый слепок текущего состояния экрана или таблицы. Данные: route, active filters, visible state, export format.

Система инициирует скачивание файла.

**Альтернативные пути:**

Пользователь выбирает Расширенный экспорт вместо прямой выгрузки.

**Где пользователь может сломаться:**

Не понимает, что быстрый экспорт выгружает именно текущий экран, а не всю систему.

**Системные состояния / ответы:**

Ошибка формирования файла должна быть показана явно.

### 22. Переход в расширенный экспорт

**Цель пользователя:** детально настроить состав выгрузки по текущему контексту.

**Предусловия:** пользователь находится внутри рабочего раздела и открыл локальный экспорт.

**Шаги:**

Пользователь выбирает Расширенный экспорт из локального меню. Система запоминает текущий контекст раздела. Данные: source module, source filters, source selection.

Система открывает модуль расширенного экспорта с уже предзаполненными параметрами из исходного контекста. Данные: export preset from source screen.

Пользователь настраивает состав выгрузки: разделы, проекты, формат, шаблон. Данные: selected sections, selected projects, format, template.

Если модуль в текущей версии представлен прототипной заглушкой, система честно это сообщает и не маскирует заглушку под рабочую функцию.

**Альтернативные пути:**

Пользователь отменяет переход и остается на исходном экране.

**Где пользователь может сломаться:**

Не понимает, что попал уже не в локальный экспорт текущего экрана, а в общий расширенный контур выгрузки.

**Системные состояния / ответы:**

loading — модуль загружается.

empty — пока не выбраны разделы или проекты.

error — параметры экспорта не загрузились.
